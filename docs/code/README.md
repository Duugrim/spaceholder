# Документация по коду (внутри хранилища Obsidian: `docs/`)

## Папки

| Папка | Содержание |
|--------|------------|
| **`reference/`** | Как устроена реализация: модули, данные, поведение подсистем. |
| **`reference/legacy/`** | Умышленно устаревшие тексты (старые ветки кода) — смотреть только для сравнения. |
| **`guides/`** | Материалы для разработки: паттерны листов, UI-дизайн. |
| **`tooling/`** | Внешние к системе вещи: LevelDB, Foundry CLI, npm-скрипты, сборка компендиумов, ручной каталог сценариев для `node`‑тестов (`DAMAGE_RESOLVER_TEST_CASES.md` — правила, `DAMAGE_RESOLVER_TEST_EXAMPLES.md` — примеры). |

## Метаданные в файлах

В начале каждого `.md` — YAML frontmatter:

- **`doc-type`**: `reference` | `guide` | `tooling` (и для legacy-док внутри `reference/legacy` по-прежнему `reference` / `status: legacy`).
- **`status`**: `current` — в целом согласовано с кодом; `verify` — сверить с актуальным `module/`; `legacy` — описание снятой/заменённой архитектуры; `partial` — PoC или незавершённый объём; `historical` — в основном журнал прошлых правок, не полноценное «как сейчас устроено».
- **`note`**: краткое пояснение, если статус не `current` (по смыслу).
- **`tags`**: `sh-code-doc/<тип>` и `sh-code-doc/status/<статус>` — удобно для графа/поиска в Obsidian.

Точку входа по репозиторию для людей и агентов по-прежнему стоит начинать с `AGENTS.md` в корне системы.
